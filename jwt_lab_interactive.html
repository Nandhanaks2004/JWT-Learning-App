Traditional: Session-Based Authentication
What happens after you log in — the old way
the big idea

key insight
The session ID in the cookie is just a random key. The actual data (who you are, your role) lives on the server. The cookie is just a lookup key.
// section 02
The Problem with Sessions
Why sessions break at scale
core problem
The server must remember every user. This creates serious problems in real-world systems.
problem 1: memory
Each session takes RAM. 10,000 users = 10,000 sessions stored. Memory fills up as users grow.
problem 2: restart
Server crashes or restarts → RAM is wiped → ALL sessions lost → every user is logged out instantly.
problem 3: scaling
Add a second server (Server B) — it has no sessions from Server A. User gets routed to B → treated as a stranger.
problem 4: sharing
Microservices need to share session data. Requires a central session store — another point of failure and complexity.

JWT: The Solution
What if the client carries its own proof of identity?
the big shift
Instead of the server remembering you, the server gives you a signed token after login. You carry it. You prove who you are on every request. The server just verifies the signature — no memory needed.
JWT = JSON Web Token
A JWT is a small text string containing three parts: user data (payload), metadata (header), and a cryptographic signature that proves the data wasn't tampered with.
# Session approach — server must remember
Server RAM: { session_id → user_data } ← grows forever

# JWT approach — client carries the data
Client stores: eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0Mn0.signature
Server only needs: secret_key to verify
session model
Client → sends cookie ID
Server → looks up session table
Server → finds user data
Server → allows/denies
JWT model
Client → sends token (with data inside)
Server → verifies signature
Server → reads data from token
Server → allows/denies
important note
JWT is encoded (base64), not encrypted. Anyone can read the payload. The signature only proves it wasn't modified — it doesn't hide the data. Never put passwords or secrets in the payload.

How JWT Works: Step by Step
Follow the complete authentication flow
step-by-step flow
Click "Next Step" to begin the JWT flow...
what happens at each stage
  • User sends credentials (username + password) to the server
  • Server checks credentials against the database
  • If valid, server creates a JWT with user info and signs it with a secret key
  • Server sends JWT back to client in the response
  • Client stores JWT (cookie or localStorage)
  • For every protected request, client adds JWT in Authorization header
  • Server receives request, verifies JWT signature using the same secret key
  • If valid, server reads user info from token and grants access — no DB lookup needed

    JWT Structure
    Three parts, separated by dots — click each to explore
    // a real JWT token:
    eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjo0Miwicm9sZSI6InVzZXIiLCJleHAiOjE3MTYyMzkwMjJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
    ■ header ■ payload ■ signature ← click each part
    header — algorithm metadata
    // decoded header:
    {
      "alg": "HS256", // algorithm: HMAC SHA-256
      "typ": "JWT"   // token type
    }
    The header tells the server which algorithm was used to create the signature. HS256 uses a shared secret key. RS256 uses public/private key pairs.
    payload — user claims (readable by anyone!)
    // decoded payload:
    {
      "user_id": 42,            // who you are
      "role": "user",          // your permissions
      "exp": 1716239022       // expires (unix timestamp)
    }
    WARNING: This is base64 encoded, NOT encrypted. Anyone who gets this token can decode and read the payload. Never store passwords, credit cards, or sensitive data here.
    signature — tamper-proof seal
    // how the signature is created:
    signature = HMAC_SHA256(
      base64(header) + "." + base64(payload),
      SECRET_KEY // only the server knows this
    )
    If an attacker changes even one character in the payload, the signature will no longer match. The server rejects any token where header+payload doesn't produce the same signature using its secret key.
    key insight: encoding vs encryption
    Base64 is just a text encoding format — like translating English to Morse code. It's not encryption. Any website (like jwt.io) can decode a JWT instantly. The only security comes from the signature.

    server-side verification logic
    # pseudocode: what the server does on each request
    def verify_jwt(token, secret_key):
      header, payload, signature = token.split(".")
      expected = HMAC(header + "." + payload, secret_key)
      if signature != expected:
        raise InvalidSignature → 401 Unauthorized
      if payload["exp"] < now():
        raise TokenExpired → 401 Unauthorized
      return payload # trust the data inside

    JWT Vulnerabilities — Attacker's View
    Think like an attacker. Understand how JWT can be exploited.
    attack 1: payload tampering
    Attacker decodes payload (it's just base64), changes role from "user" to "admin", re-encodes it, and sends it.
    # original payload
    { "user_id": 42, "role": "user" }

    # attacker modifies it
    { "user_id": 42, "role": user → "admin" }

    attack 2: weak secret key
    If the server uses a weak secret key like "secret" or "password123", an attacker can brute-force it offline and then forge any token.

    attack 3: token theft via XSS
    If JWT is stored in localStorage, any JavaScript on the page can access it — including injected malicious scripts.

    Best Practices
    How to use JWT securely in production
    security checklist
    ✓

    Use a strong, random secret key

    Minimum 256 bits of entropy. Never use predictable values like "secret", your app name, or short strings. Use a proper random generator: openssl rand -hex 32

    ✓

    Always validate the signature

    Never skip verification. Never trust the payload before confirming the signature matches. Accept only algorithms you explicitly allow — never accept "alg: none".

    ✓

    Set a short expiration (exp claim)

    Short-lived tokens (15 minutes to 1 hour) limit damage if stolen. Use refresh tokens for maintaining sessions without long-lived access tokens.

    ✓

    Store JWT in httpOnly cookies

    HttpOnly cookies cannot be accessed by JavaScript — XSS attacks cannot steal them. Pair with SameSite=Strict and Secure flags for full protection.

    ✓

    Never put sensitive data in the payload

    The payload is just base64 encoded — anyone can read it. Only store non-sensitive identifiers: user ID, role, expiration. No passwords, SSNs, or PII.

    # secure JWT configuration example
    jwt.sign(payload, SECRET_KEY, {
      algorithm: "HS256",         ✓ explicit algorithm
      expiresIn: "15m",         ✓ short expiry
      issuer: "myapp.com",       ✓ identify the issuer
    })
