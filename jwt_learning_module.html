01

What is JWT?

Imagine you walk into a concert venue. You pay at the gate, get a wristband, and from that point on — no one asks you to pay again. Every security guard at every door just glances at your wristband and lets you through.

JWT works exactly the same way on the internet. You log in once with your password. The server gives you a token — a wristband made of data. For every action you take after that, you show the wristband instead of repeating your password.

One-line definition

JWT = A tamper-proof digital wristband that proves you are who you say you are, without the server needing to remember you.

JWT stands for JSON Web Token. It is an open standard (RFC 7519) that defines a compact, self-contained way to securely transmit information as a JSON object.

Three words that matter

  • Compact  Small enough to be sent in a URL, header, or cookie.
  • Self-contained  The token itself carries all the user information needed.
  • Signed  Cannot be tampered with — any modification breaks the signature.
02

Why JWT Exists

To understand why JWT was invented, you need to first understand what it replaced — and why the old way broke down.

The Old Way — Session-Based Authentication

When you logged into a website using the old session method, here is what happened behind the scenes:

  1. You submit your username and password.
  2. The server creates a session record in its memory (or database): session_id=abc123 → user=john
  3. The server sends your browser a cookie containing just the session ID.
  4. Every time you click something, your browser sends that cookie.
  5. The server has to look up session_id=abc123 in its storage to figure out who you are.

This works fine for a single small website. But as soon as systems grow, serious problems appear.

❌ Session-Based — Problems
  • Server must store a record for every logged-in user
  • If the server restarts, all sessions are wiped — everyone is logged out
  • Hard to scale: two servers don't share each other's session memory
  • Each request forces a database lookup — adds latency
  • Doesn't work cleanly with mobile apps or third-party APIs
  • Stateful: the server is burdened with remembering state
✅ JWT-Based — Solutions
  • Server stores nothing — the token is self-contained
  • Server restarts are harmless — the token lives with the client
  • Any server with the secret key can verify any JWT instantly
  • No database lookup on every request — just signature math
  • Works with any client: browsers, mobile, microservices, IoT
  • Stateless: the token is the proof
Aspect Session-Based JWT-Based
Where state livesServer memory / databaseInside the token (client-side)
ScalabilityHard (sticky sessions needed)Easy (any server can verify)
Works across domainsNo (cookie restrictions)Yes (sent in header)
RevocationEasy (delete the session)Harder (requires extra logic)
Good for mobile/APIsNoYes
Server memory useGrows with usersZero

The Passport Analogy

The best real-world object that maps to JWT is a passport. Let's map every detail.

Passport vs JWT
Passport Feature JWT Equivalent
Issued by a governmentIssued by the server after successful login
Contains your name, nationality, birth dateContains user ID, role (admin/user), expiry time
Has an official stamp / signatureHas a cryptographic signature
Any border officer can verify it (no phone call needed)Any server with the secret key can verify the JWT instantly
Expires after 10 yearsJWT expires after minutes, hours, or days
Alteration voids it — tamper-evident paperAltering one byte breaks the signature
You carry it — the government doesn't follow you aroundToken stored on the client — server stores nothing

Notice the critical insight: a border officer does not call the passport office to verify your passport. They just look at it, check the signature, and verify it is authentic. This is why JWT is so fast and scalable — verification is purely mathematical, not a database lookup.

The key distinction: Proof vs. Memory

With sessions, the server remembers you. With JWT, you prove yourself. This single shift in responsibility — from server to token — is what makes modern distributed systems possible.

Let's walk through each step in detail:

Step 1 — User submits credentials

The user enters their email and password. This is the only time the password is used. It is sent over HTTPS to the server. The password itself is never stored anywhere after this point.

Step 2 — Server authenticates the user

The server looks up the user in the database, compares the hashed password, and confirms the user is real. If credentials are wrong — no token is issued and the request is rejected immediately.

Step 3 — Server creates and signs the JWT

The server bundles user information (user ID, role, expiry time) into a JSON object, then cryptographically signs it using a secret key. The token is now ready to be issued. The server does NOT store this token anywhere.

Step 4 — Token is sent to the client

The server sends the JWT back in its HTTP response. The client — whether a browser, mobile app, or another service — receives and stores it.

Step 5 — Client sends token with every request

For every subsequent API call, the client attaches the JWT inside the Authorization header as a Bearer token. No password is sent again.

Step 6 — Server verifies the signature

The server receives the request, extracts the JWT, and mathematically verifies the signature using its secret key. If the signature is valid and the token hasn't expired — access is granted. If anything is wrong — access is denied immediately.

05

JWT Lifecycle

A JWT is not permanent. It has a birth, a useful life, and a death. Understanding this lifecycle is crucial for both development and security.

🔑 Login — Token is Born
User authenticates. Server creates a signed JWT with an iat (issued-at) and exp (expiry) timestamp baked into the payload. Token is sent to the client.
🚀 Active Use — Token in Flight
The client attaches the token to every API request in the Authorization header. The server verifies the signature and expiry on each request. This phase typically lasts minutes to hours.
🔄 Refresh (Optional) — Token Renewed
Before the access token expires, many systems issue a long-lived refresh token. The client uses this to get a new access JWT without forcing the user to log in again. Access tokens stay short-lived; refresh tokens are stored securely.
⏰ Expiry — Token Dies
When the exp timestamp passes, the token is no longer valid. Any request with an expired token receives a 401 Unauthorized response.
🔁 Re-Login — Cycle Repeats
Without a refresh token (or when the refresh token itself expires), the user must log in again. A new JWT is issued and the cycle starts over.

Why expiry is a security feature

If a JWT is stolen, the attacker can only use it until it expires. A short expiry (15–60 minutes) limits the damage window significantly. This is why never setting an expiry time is one of the most dangerous JWT mistakes.

JWT Structure

Every JWT, without exception, is made of exactly three parts, separated by dots. When you see a JWT, it looks like a long string of random characters — but it has precise internal structure.

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEwMSwicm9sZSI6ImFkbWluIiwiZXhwIjoxNzA4MzUzNjAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Each part is Base64URL-encoded. Let's decode all three.

Part 2 — Payload

Part 3 — Signature

Part 1 — Header

The header is a JSON object that tells the receiver two things: what type of token this is, and which algorithm was used to sign it.

{
  "alg": "HS256",   // Signing algorithm (HS256 or RS256)
  "typ": "JWT"    // Always "JWT"
}

Security note: The algorithm field is critical. An attacker can try to change "alg":"none" to bypass signature checking entirely — a famous JWT attack (covered in section 9).

Part 2 — Payload (Claims)

The payload is where the actual user data lives. Each piece of data is called a claim.

{
  "sub":    101,            // Subject — the user ID
  "email": "john@acme.com", // Custom claim
  "role":  "admin",         // User's role/permission level
  "iat":   1708350000,     // Issued at (Unix timestamp)
  "exp":   1708353600      // Expires at (1 hour later)
}

✅ Readable

The payload is only Base64-encoded, NOT encrypted. Anyone who has the token can decode and read the payload.

❌ Do NOT store secrets here

Never put passwords, credit card numbers, or truly sensitive data in the payload. It is visible to anyone holding the token.

Part 3 — Signature (The Trust Engine)

The signature is the most important part of JWT. It is what makes the whole system trustworthy. Here's exactly how it is created:

// Pseudocode — how the signature is generated
signature = HMAC_SHA256(
  base64url(header) + "." + base64url(payload),
  SECRET_KEY
)

The server takes the encoded header and payload, appends the secret key that only it knows, and runs the whole thing through a hashing algorithm. The output is the signature.

Why tampering is impossible

If an attacker changes "role":"user" to "role":"admin" in the payload, the signature check will fail. The server will recompute the expected signature and it will not match the one in the token. The request will be rejected. The attacker cannot forge a valid signature without knowing the secret key.

PartEncoded?Encrypted?Readable?Tamper-proof?
Header Base64URL No Yes Yes (via signature)
Payload Base64URL No Yes Yes (via signature)
Signature Base64URL N/A (hashed) No (hash) Yes — it IS the proof

HS256 vs RS256 — Which Algorithm?

AlgorithmKey TypeWho signsWho verifiesBest for
HS256 One shared secret Server Any service with the secret Single-server apps
RS256 Private key (sign) + Public key (verify) Auth server (private key) Any service (public key) Microservices, enterprise

RS256 is considered more secure for distributed systems because the secret used to sign tokens never needs to leave the auth server — other services only need the public key to verify.

07

Using JWT in Requests

After login, every protected API request must carry the JWT. The standard way to do this is via the Authorization HTTP header with the Bearer scheme.

// Making a protected API request with JWT
GET /api/user/profile HTTP/1.1
Host: api.acme.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEwMSwicm9sZSI6ImFkbWluIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
Content-Type: application/json

What is "Bearer"?

The word Bearer means "the person holding this token." It signals that whoever presents this token is authorized — the server does not care who they are beyond what's in the token. This is precisely why stolen tokens are so dangerous: the server treats anyone holding a valid token as the legitimate user.

// Server-side JWT verification (Node.js example)
const jwt = require('jsonwebtoken');

function verifyToken(req, res, next) {
  const authHeader = req.headers['authorization'];
  const token = authHeader?.split(' ')[1]; // Extract after "Bearer "

  if (!token) return res.status(401).json({ error: 'No token provided' });

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded; // Attach decoded payload to request
    next();
  } catch (err) {
    res.status(403).json({ error: 'Invalid or expired token' });
  }
}
08

Token Storage

Where you store the JWT on the client side has major security implications. There is no option without some risk — the key is choosing the right trade-off for your threat model.

localStorage

Persists across browser sessions. Easy to access with JavaScript.

High Risk

Any JavaScript on the page — including injected scripts from XSS attacks — can read localStorage. A single XSS vulnerability = full token theft.

sessionStorage

Cleared when the tab is closed. Slightly better than localStorage.

High Risk

Still vulnerable to XSS. An attacker who can run JavaScript can steal the token before the session ends.

HttpOnly Cookie

The browser handles it automatically. JavaScript cannot access it.

Recommended

Immune to XSS token theft since JavaScript can't read it. Requires CSRF protection (use SameSite=Strict or anti-CSRF tokens).

Memory (JS Variable)

Stored in app state. Cleared on page refresh.

Medium Risk

Most secure against XSS but impractical — users must log in again after every refresh. Used with short-lived access tokens + HttpOnly refresh tokens.

Security Rule of Thumb

Store access tokens in memory. Store refresh tokens in HttpOnly cookies. This gives you both XSS resistance (no JS access to refresh token) and CSRF resistance (short-lived access token means any stolen CSRF request expires quickly).

09

Common JWT Attacks

Most JWT vulnerabilities are not flaws in the JWT standard itself — they are developer mistakes in how the standard is implemented. Here are the attacks every security student must know.

Attack How It Works What Attacker Achieves Severity
alg=none Attack Attacker changes the header's algorithm to none, then strips the signature. A vulnerable server skips signature validation entirely. Full authentication bypass — any user ID can be claimed Critical
HS256/RS256 Confusion When an app uses RS256, the public key is publicly known. Attacker changes algorithm to HS256 and signs the token using the public key as the "secret." A confused server accepts it. Forge tokens for any user including admin Critical
Weak Secret Key Attacker brute-forces or guesses the HMAC secret key (e.g. "secret", "password123") using tools like hashcat or jwt_tool. Can sign arbitrary tokens with any claims Critical
Missing Expiry (no exp) Developer forgets to set the exp claim. The token is valid forever. A stolen token works indefinitely — no expiry window High
Token Theft (XSS) Token stored in localStorage. Attacker injects JavaScript through an XSS vulnerability. Script reads localStorage.getItem('token') and sends it to attacker's server. Full account takeover without knowing the password High
No Signature Verification Developer decodes the payload to get user info but never actually verifies the signature. Any JWT — real or forged — is accepted. Complete authorization bypass Critical
Sensitive Data in Payload Developer stores passwords, SSNs, or secrets in the payload. Since payload is only Base64-encoded, anyone who intercepts the token can decode it. Data exposure even with valid tokens Medium
Insecure Transmission JWT sent over HTTP instead of HTTPS. Network attacker intercepts the token in transit (MITM). Token theft without any code exploitation needed High

The Attacker Mindset

A security professional must be able to think like an attacker. When an attacker sees a JWT in an application, here is exactly what goes through their mind — step by step.

⚡ Attacker's Mental Checklist
Step 1 — Capture the token

Intercept the JWT from browser storage (DevTools → Application → localStorage), HTTP traffic, or error messages that accidentally leak it.

Step 2 — Decode the payload

Paste the token into jwt.io or run echo '[payload]' | base64 -d to read all claims. Look for interesting fields: role, admin, userId, exp.

Step 3 — Try alg=none

Change the algorithm in the header to "alg":"none", modify the payload (e.g. escalate role to admin), remove the signature, and re-encode. If the server doesn't enforce algorithm, the forged token is accepted.

Step 4 — Try to crack the secret

Use hashcat with a wordlist: hashcat -m 16500 token.txt rockyou.txt. If the secret is weak (e.g. "secret", "jwt", "password"), it can be cracked in seconds.

Step 5 — Check for missing verification

Modify the payload without changing the signature. If the server still accepts the modified token, it is not verifying the signature at all — one of the worst possible JWT bugs.

What attackers actually target

The payload's role field is the most valuable target — changing "role":"user" to "role":"admin" is the dream. The userId or sub field is the second target — changing it lets the attacker impersonate any other user (IDOR via JWT). The exp field is also interesting — changing it to a future date could extend a stolen token's validity if the server trusts the payload instead of enforcing server-side time checks.

When JWT Fails

JWT is not a silver bullet. It has genuine architectural limitations that every developer and security engineer must understand.

Problem 1 — You Cannot Instantly Revoke a JWT

This is the biggest weakness of JWT's stateless design. Because the server stores nothing, there is no way to "delete" a token. If a user logs out, you can delete the token from their browser — but if an attacker already copied it, their copy still works until it expires.

Workarounds: Maintain a token blocklist (a Redis set of revoked JTI claims). Or use very short expiry times (15 minutes) with refresh tokens. Both approaches partially re-introduce state — a trade-off that must be consciously made.

Problem 2 — Logout is a Lie (Without Extra Work)

When you call POST /logout and the server just tells the client to delete its token — the token itself is still valid on the server side. A malicious client (or attacker) can keep using it. True server-side logout requires token revocation infrastructure.

Real logout needs: Deleting the token client-side AND adding the token's unique ID (jti claim) to a server-side blocklist.

Problem 3 — Over-Trusting the Payload

A dangerous anti-pattern: developers write code like if (token.role === 'admin') without first verifying the signature. If signature verification is skipped or done incorrectly, any attacker who modifies the payload bypasses all access control.

Rule: Always verify the signature first. The payload is meaningless until the signature is confirmed valid.

Problem 4 — Token Size Growth

JWT grows with every claim added to the payload. In high-traffic APIs, hundreds of megabytes of bandwidth per day can be spent transmitting bloated tokens. Keep payloads lean — only store what is necessary for authorization decisions.

12

Best Practices

Follow these practices and you will avoid the vast majority of JWT vulnerabilities found in production applications.

Always verify the signature

Use a battle-tested JWT library. Never decode the payload manually without first calling verify(). Never trust the payload on faith.

Always set an expiry time

Access tokens: 15–60 minutes. Never issue tokens without exp. Use refresh tokens for persistence instead of long-lived access tokens.

Use a strong secret key

At minimum 256 bits of randomness for HS256. Generate with: openssl rand -base64 64. Never use dictionary words.

Prefer RS256 for distributed systems

If multiple services need to verify tokens, RS256 lets them verify with the public key — without ever exposing the private signing key.

Whitelist allowed algorithms

Explicitly specify which algorithm your server accepts. Reject tokens with "alg":"none". Never use dynamic algorithm selection from the token itself.

Store securely on client

HttpOnly cookies for refresh tokens. Memory-only for access tokens in SPAs. Never store sensitive tokens in localStorage.

Transmit over HTTPS only

Add Secure flag to cookies. Enforce HTTPS on all endpoints. A token transmitted over HTTP can be stolen by any network observer.

Implement a token blocklist

Use Redis to store revoked jti values for critical operations (logout, password change, account compromise). Check on every request.

Keep payload minimal

Only include what is needed for authorization. No passwords, no secrets, no PII beyond what is strictly necessary. Remember: payload is readable by anyone with the token.

13

Summary — Key Takeaways

If you remember nothing else from this module, remember these five things.

🎫
JWT is a signed digital passport. After login, the server issues a cryptographically signed token. The client presents this token with every request instead of resending the password. The server verifies the signature — no database lookup needed.
🔒
Trust comes from the signature, not the payload. The payload is readable by anyone. The signature — created from Header + Payload + Secret Key — is what makes the token tamper-proof. Always verify the signature before trusting any claim.
⏰
Expiry is a security feature, not an inconvenience. Short-lived tokens limit the damage window when stolen. Always set exp. Pair with refresh tokens for seamless user experience without sacrificing security.
⚠️
Most JWT attacks exploit implementation mistakes. alg=none, weak secrets, missing verification, no expiry, and storing tokens in localStorage account for the overwhelming majority of JWT vulnerabilities found in real applications.
🚫
JWT has trade-offs. Stateless design means instant revocation is impossible by default. True logout requires a blocklist. Over-trusting the payload without signature verification is a critical flaw. Use JWT where it fits — not as a universal solution for all authentication needs.
The Complete Picture in One Paragraph

JWT replaced server-side sessions by moving the authentication state from the server to the client. A signed token — containing user identity, role, and expiry — is issued at login and presented with every request. The three-part structure (Header.Payload.Signature) ensures the token can be read by anyone but only forged by someone who knows the secret key. Its stateless nature makes it ideal for distributed systems and APIs, but creates genuine challenges for revocation. Security depends almost entirely on correct implementation: strong secrets, proper signature verification, short expiry, and secure storage.
